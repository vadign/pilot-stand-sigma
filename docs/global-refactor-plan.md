# План глобального рефакторинга Sigma

Дата: 2026-04-02

## 1) Что видно по текущему состоянию

### Архитектурные сигналы
- `src/App.tsx` одновременно отвечает за маршрутизацию, lazy-импорты и legacy-redirect, что затрудняет масштабирование роутинга.
- `src/features/pages/MayorDashboardPage.tsx` содержит слишком много ответственности: фильтрацию, агрегации, синхронизацию query-параметров, подготовку карточек/метрик, условный рендеринг разных подсистем.
- `src/components/Layout.tsx` связан сразу с несколькими фичами (`ask-sigma`, `live`), что повышает связность базового shell-слоя.

### Дубли и кандидаты на зачистку
- Есть два barrel-файла для страниц (`src/features/pages.tsx` и `src/features/pages/index.ts`), при этом реальный импорт страниц в `App.tsx` идет напрямую по файлам.
- Маршруты `/resources`, `/reports`, `/settings` используют общий `PlaceholderPage` и выглядят как временные/демо-разделы.
- Существует legacy-маршрут `/public-transport`, который только редиректит на `/mayor-dashboard` через параметры.
- В `src/assets` лежат `react.svg` и `vite.svg`; по коду они не используются.

## 2) Цели рефакторинга

1. **Повысить читаемость**: сократить размер «толстых» компонентов, перенести вычисления в селекторы/хуки/презентеры.
2. **Снизить связность (coupling)**: изолировать доменные подсистемы и уменьшить cross-feature импорты.
3. **Удалить устаревшие/неиспользуемые сущности**: убрать dead routes, legacy-redirects, дублирующие barrel-файлы, неиспользуемые ассеты/утилиты.
4. **Подготовить основу под дальнейшее развитие**: понятная модульная структура и контрактные границы между слоями.

## 3) Целевая структура (высокоуровнево)

- `app/` — корневой shell (роутер, провайдеры, layout composition).
- `pages/` — только page composition и wiring.
- `widgets/` — крупные UI-блоки, используемые страницами.
- `features/` — пользовательские сценарии и бизнес-операции.
- `entities/` — доменные модели, селекторы, типы.
- `shared/` — общие UI, утилиты, инфраструктура.

> Это можно внедрять инкрементально без big-bang миграции.

## 4) План по этапам

### Этап A — Аудит и фиксация baseline (1–2 дня)
- Снять карту маршрутов: реальные, legacy, placeholder.
- Построить dependency-graph по `src` и выявить циклы/слои с максимальной связностью.
- Сформировать реестр кандидатов на удаление (файлы, роуты, экспорты, ассеты).
- Зафиксировать baseline-метрики: размер файлов, количество cross-feature импортов, время сборки, покрытие тестами.

**Критерий завершения**: есть документированный список проблем + приоритизация на удаление/перенос.

### Этап B — Роутинг и навигация (2–3 дня)
- Вынести роут-конфигурацию из `App.tsx` в модуль `app/router` (конфигурация + mapping lazy components).
- Формализовать статус маршрутов: `active | experimental | deprecated`.
- Для `deprecated`:
  - либо удаление,
  - либо ограниченный по сроку redirect с датой удаления.
- Принять решение по `/resources`, `/reports`, `/settings`:
  - удалить из меню/роутера,
  - или оформить как feature-flagged разделы.
- Определить срок удаления legacy `/public-transport` redirect.

**Критерий завершения**: единый источник истины по маршрутам + отсутствуют «безвременные» placeholder/legacy маршруты.

### Этап C — Декомпозиция Mayor Dashboard (4–6 дней)
- Разбить `MayorDashboardPage` на:
  - `useMayorDashboardState` (query params, режимы, selected filters),
  - `useMayorDashboardMetrics` (агрегации/метрики),
  - `MayorDashboardView` (чистый презентационный компонент),
  - подкомпоненты секций (`PriorityPanel`, `MetricsGrid`, `DistrictPanel`, и т.д.).
- Вынести доменные вычисления из компонента в селекторы/презентеры в `entities/incidents` или `features/mayor-dashboard/model`.
- Добавить тесты на вычисления отдельно от UI.

**Критерий завершения**: page-компонент ≤ 200–250 строк, вычисления покрыты unit-тестами.

### Этап D — Развязка Layout и feature boundaries (2–3 дня)
- Убрать feature-специфическую логику из общего `Layout`:
  - вынести `AskSigmaBar`/`AnswerPanel` в отдельный `widget`-слой оболочки,
  - инициализацию live bootstrap перенести в app-level provider/effect.
- Ограничить доступ компонентов shell к store фич напрямую.

**Критерий завершения**: `Layout` остается инфраструктурным, не доменным компонентом.

### Этап E — Удаление legacy/dead code (2–4 дня)
- Удалить дублирующийся barrel (оставить один вариант и унифицировать импорты).
- Удалить неиспользуемые ассеты (`src/assets/react.svg`, `src/assets/vite.svg`) и прочие файлы из реестра dead code.
- Очистить неиспользуемые экспорты/утилиты по результатам статического анализа.
- Прогнать тесты и линтер после каждого батча удалений.

**Критерий завершения**: нет неиспользуемых файлов/экспортов в согласованном scope.

### Этап F — Укрепление контрактов и guardrails (2 дня)
- Ввести правила линтинга на импорт слоев (например: page -> widget -> feature -> entity -> shared).
- Добавить проверку «мертвых экспортов» в CI.
- Ввести шаблон ADR для решений по деприкации роутов и удалению модулей.

**Критерий завершения**: новые нарушения архитектурных границ блокируются CI.

## 5) Что удалять в первую очередь (кандидаты)

1. Дублирующие barrel-экспорты страниц (`src/features/pages.tsx` vs `src/features/pages/index.ts`) — оставить единый вход.
2. Placeholder-маршруты без roadmap (`/resources`, `/reports`, `/settings`) — удалить или скрыть за флагом.
3. Legacy redirect `/public-transport` после периода обратной совместимости.
4. Неиспользуемые ассеты в `src/assets`.

## 6) Риски и как их снять

- **Риск:** сломать внешние deep-links.
  - **Снижение:** таблица редиректов + e2e smoke на все публичные пути.
- **Риск:** регресс в расчетах dashboard.
  - **Снижение:** snapshot/contract tests на вычисления метрик и приоритетов.
- **Риск:** затяжная миграция структуры.
  - **Снижение:** инкрементальные PR по этапам, каждый этап — отдельный deployable slice.

## 7) Практический порядок PR

1. PR-1: Route registry + маркировка deprecated/placeholder.
2. PR-2: Декомпозиция MayorDashboardPage (state/model/view).
3. PR-3: Layout boundary cleanup.
4. PR-4: Dead code cleanup (роуты, barrel, ассеты, экспорты).
5. PR-5: Архитектурные линт-ограничения + CI checks.

## 8) Метрики успеха

- Средний размер page-компонента уменьшен минимум на 30%.
- Количество cross-feature импортов снижено минимум на 40%.
- Количество deprecated/placeholder маршрутов в main-роутере = 0 (или документированно ограничено).
- Dead exports/files по agreed tooling = 0.
